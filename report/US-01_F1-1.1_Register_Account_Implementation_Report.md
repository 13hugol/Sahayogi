# US-01 / F1-1.1 - Register Account Implementation Report

## User Story

As a new user, I want to register an account so that I can access the skill exchange platform.

## Completion Status

Completed for the current backend scope on the `main` branch. Registration is implemented with mandatory fields, duplicate-email checks, password hashing, profile-shell creation, role assignment, CSRF protection, and email-verification token generation.

## Implementation Summary

- `app/controllers/auth_controller.py` validates submitted registration data before creating an account.
- `app/models/user.py` creates the user record, hashes the password, assigns the default `user` role, and creates the profile shell.
- `app/utils/tokens.py` generates signed email-verification tokens.
- `app/utils/email.py` sends email through SMTP when configured and logs locally to `instance/mail.log` when SMTP is unavailable.
- `app/templates/auth/register.html`, `verify_email_pending.html`, and `resend_verification.html` provide the user-facing registration and verification flow.
- `app/routes/auth_routes.py` maps `/auth/register`, `/auth/verify-email/<token>`, and `/auth/resend-verification`.

## Acceptance Criteria Evidence

| Acceptance Criterion | Implementation Evidence |
| --- | --- |
| Registration requires full name, email, password, and location. | The registration controller reads and validates `full_name`, `email`, `location`, `password`, and `confirm_password`. |
| Email format and duplicate email are validated. | The controller rejects malformed emails and checks `User.find_by_email()` before creating an account. |
| User receives a verification email and must confirm. | A signed token is generated, stored on the user record, and sent or logged through `send_email()`. The `/auth/verify-email/<token>` route marks the email as verified. |
| Successful registration sends the user onward with a welcome/pending message. | The registration flow renders `auth/verify_email_pending.html` and flashes a clear account-created message. |

## Database and Security Notes

- `users` stores identity, password hash, email verification state, failed login counters, status, and role.
- `profiles` stores the profile shell created during registration.
- Parameterized SQL is used through the `Database` helper.
- Passwords are hashed with Werkzeug secure password hashing before storage.
- CSRF validation runs before every POST request unless disabled in tests.
