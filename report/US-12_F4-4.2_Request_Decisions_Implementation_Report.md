## Completion Status

Completed on the `Aayan` branch. The system now supports a full request management system allowing users to view incoming pending requests, accept or decline them, record decline reasons, view sent requests, and cancel pending sent requests. Accepting a request automatically initializes a messaging conversation thread and triggers notifications.

## Implementation Summary

- **Database Updates**: Added two tables in `app/database.py`:
  - `exchange_requests`: Stores incoming learner requests, including listing links, learner details, message details, status, and decline reasons.
  - `exchanges`: Stores the active exchange instances initialized after accepting requests.
- **Model Classes**:
  - `ExchangeRequest` in `app/models/exchange_request.py` with properties resolving sender name, recipient, and details.
  - `Exchange` in `app/models/exchange.py`.
  - Exposed them in `app/models/__init__.py`.
- **Repositories**:
  - `ExchangeRequestRepository` in `app/repositories/exchange_request_repository.py` to handle CRUD operations on requests, listing pending and historical requests.
  - `ExchangeRepository` in `app/repositories/exchange_repository.py` to manage active exchanges.
- **Controllers & Routing**:
  - Wired up methods for inbox loading, sent list loading, request creation, acceptance, declination with reasoning, and cancellation in `FrontendController` (`app/controllers/frontend_controller.py`).
  - Registered request endpoints in `app/routes/frontend_routes.py`.
- **UI Pages**:
  - Rendered request lists dynamically using Jinja templates in `app/templates/requests/inbox.html` and `sent.html`.
  - Included interactive elements for forms validating and submitting accept, decline, and cancel actions.

## Acceptance Criteria Evidence

| Acceptance Criterion | Implementation Evidence |
| --- | --- |
| Incoming requests appear in a Requests inbox with sender's name, skill requested, and message. | Implemented in `app/templates/requests/inbox.html` where incoming requests fetch the sender's full name, requested listing title, and the custom requested message. |
| Accept and Decline buttons are present on each request; action triggers an in-app notification to sender. | Actions bind to `accept` and `decline` controller routes. Accepting/declining triggers `Notification.create()` to the learner/sender. |
| Declining a request moves it to declined history with an optional reason message. | Declining updates status to `'declined'`, saves `decline_reason` if provided in the POST body, and moves the request to the History section of the Request Inbox. |
| Accepting a request creates an exchange record and opens a messaging thread between both users. | Accepting inserts into the `exchanges` table, marks status as `'accepted'`, and invokes `MessageService.create_conversation` with permission source `'accepted_exchange'`. |

## Database and Security Notes

- **Access Controls**: Actions verify ownership of the target listing before accepting/declining. Only the requesting user (learner) can cancel a pending request.
- **Credit Validation Check**: Learners cannot request a credit-based listing if their available credit balance is less than the credit cost of the listing. In Sprint 3, a mock starting balance of `100` credits is configured for all users.
- **CSRF Protection**: All POST request forms validate CSRF tokens via Jinja.
