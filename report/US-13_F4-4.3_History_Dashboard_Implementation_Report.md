As a user, I want a history dashboard so that I can track all my past and ongoing skill exchanges.

## Completion Status

Implemented on the `Chayanga` branch for the current Flask/Jinja backend scope. The story now has a persisted exchange-history item model, user-scoped repository queries, status/date filtering, dashboard summary wiring, exchange detail lookup, and visible links to listings, message threads, and review actions.

## Implementation Summary

- Added `ExchangeHistoryItem` as the model for history dashboard rows with listing, partner, role, status, timestamps, conversation, and review-submitted state.
- Added the `exchange_history_items` database table with user ownership, status/date-friendly indexes, and cascade cleanup when a user is deleted.
- Added `ExchangeHistoryRepository` with parameterized SQL for user-scoped list, count, and detail lookup operations.
- Added `ExchangeHistoryService` for allowed status options, date parsing, reversed-date normalization, and dashboard query construction.
- Updated the main dashboard to show the user's recent exchange history and total exchange count.
- Updated `/exchanges/` to render all user history rows with status filtering, date-range filtering, visible timestamps, partner/role details, and links to the exchange detail, original listing, and associated message thread.
- Updated completed exchange rows so `Leave a Review` appears only when the history item has no submitted review; reviewed completed exchanges show a submitted-review badge.
- Updated `/exchanges/<id>` to load only history items owned by the signed-in user before falling back to the existing frontend shell.

## Acceptance Criteria Evidence

| Acceptance Criterion | Implementation Evidence |
| --- | --- |
| Dashboard lists all exchanges with status: pending, active, completed, or declined; timestamps are visible. | `ExchangeHistoryService.VALID_STATUSES` exposes the required statuses; `exchanges/index.html` renders each row's status plus created, updated, completed, and declined timestamps when available. |
| Completed exchanges show a `Leave a Review` button if no review has been submitted yet. | `ExchangeHistoryItem.can_review` returns true only for completed rows without `review_submitted`; the dashboard template shows the button only for that state and tests cover both missing-review and submitted-review cases. |
| User can filter history by status or date range to find specific exchanges quickly. | `FrontendController.exchanges()` builds a query from `status`, `start_date`, and `end_date`; the repository applies those filters with parameterized SQL, and service tests verify status/date filtering and reversed-date normalization. |
| Each history entry links to the original listing and the associated message thread. | `exchanges/index.html` renders listing links through `listings.detail` when `listing_id` exists and message-thread links through `messages.detail` when `conversation_id` exists. |

## Database and Security Notes

- All exchange history reads are scoped by `user_id`; detail lookup requires both signed-in user ID and exchange history ID.
- Repository queries use `%s` placeholders and pass parameters separately.
- The implementation does not change request acceptance, messaging persistence, review submission logic, credit transfers, or any other exchange-adjacent feature.
- Review and message formats link into the existing frontend-shell routes; this story only controls when those links appear from the history dashboard.
- The temporary Excel lock file `Documents/~$Sahayogi_userstry (1).xlsx` was not part of the implementation and should not be committed.
