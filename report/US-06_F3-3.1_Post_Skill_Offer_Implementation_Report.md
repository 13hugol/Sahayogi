# US-06 / F3-3.1 - Post Skill Offer Implementation Report

## User Story

As a registered user, I want to post a skill offer listing so that others can find and request my services.

## Completion Status

Completed on the `Bhugol` branch. The system now allows users to post, edit, delete, and view their skill offer listings. These listings default to a "pending" status, requiring admin approval before they appear publicly in the marketplace.

## Implementation Summary

- **Database Updates**: Added `categories` and `skills` tables in `app/database.py`. Seeded default categories (`Tech`, `Music`, `Language`, `Kitchen`) using a custom Flask command in `app/commands.py`.
- **Model Classes**: Created `Category` and `Skill` models in `app/models/skill.py` to represent skill categories and listings.
- **Repositories**: Implemented `CategoryRepository` and `SkillRepository` in `app/repositories/skill_repository.py`.
- **Services**: Built `SkillService` in `app/services/skill_service.py` to manage skill listing creation, editing, deletion, and retrieval.
- **Frontend Controller**: Updated `FrontendController` in `app/controllers/frontend_controller.py` to support routes like `/listings/`, `/listings/create`, `/listings/mine`, `/listings/<id>`, `/listings/<id>/edit`, `/listings/<id>/delete`, and the search API `/listings/api/search`.
- **Routes Integration**: Wired listings routes in `app/routes/frontend_routes.py`.

## Acceptance Criteria Evidence

| Acceptance Criterion | Implementation Evidence |
| --- | --- |
| Create listings with title, description, category, exchange type, and availability. | Users fill out a form containing these fields. The title, description, category, exchange type, and availability are fully validated. |
| Retrieve offered skills from the user's profile to pre-fill choices. | The create form pre-fills the skill drop-down exclusively with the current user's offered skills (from `profile_skills` where type is `offered`). |
| Set credit costs based on exchange type. | For `credit` exchange type, the `min_credits` input determines the cost. For `barter` and `teach` types, the cost is set to `0`. |
| Limit edit and delete actions to the listing owner. | The controller checks if the logged-in user matches the owner of the listing, returning `403 Forbidden` if unauthorized. |
| Restrict visibility of unapproved listings to the owner. | Public marketplace queries filter for `approved` status, while the owner's listings route (`/listings/mine`) returns all status values. |

## Database and Security Notes

- **Foreign Keys**: The `skills` table links to `users`, `categories`, and `profile_skills` with proper cascading deletes.
- **Validation**: Server-side checks are applied to text lengths, non-negative credits, and valid foreign key mappings.
- **CSRF Protection**: Standard CSRF tokens are verified across all POST requests.
- **Authentication**: Protected listings routes redirect anonymous users to the login page.
