## Completion Status

Completed for both backend and frontend ledger interfaces. Credits are an **internal-only** system with no monetary value: they cannot be bought, sold, topped up, or withdrawn, in line with `guide.md` Â§2 (the platform must exclude real-money payments, payment gateways, deposits, and withdrawals). Users start with 100 default credits. Credit holds (reservations) are automatically managed during the skill-request lifecycle. When both participants mark an exchange complete, credit settlement runs as a single atomic repository transaction: the learner's hold is cleared, the learner is debited, the teacher is credited, both ledger rows are written, and completed-exchange counts increment together. Settlement is idempotent, so a double-submit by either user cannot move credits twice.

## Implementation Summary

- `app/database.py` defines schema structures and `ALTER TABLE` migrations for `users.credit_balance`, `credit_transactions`, and `credit_holds`.
- `app/models/credit_transaction.py` defines `CreditTransaction` and `CreditHold` models.
- `app/models/user.py` loads `credit_balance` and calculates `available_credit_balance` by subtracting active holds.
- `app/repositories/credit_repository.py` owns all credit data rules and SQL:
  - `create_hold()` reserves the listing cost on request creation.
  - `has_active_hold_for_request()` guards against a duplicate active hold if the create-request form is double-submitted.
  - `release_hold()` / `clear_hold()` release or settle holds.
  - `exchange_already_settled()` and **`settle_exchange_credits()`** perform atomic completion settlement in one database transaction: clear the hold, debit the learner, credit the teacher, write both double-entry ledger rows, and increment each participant's completed-exchange count. The method is idempotent and re-checks settlement inside the transaction.
  - `get_history_for_user()` and `get_active_holds_for_user()` feed the ledger page.
- `app/controllers/frontend_controller.py`:
  - `wallet()` retrieves history and active holds to render `ledger.html`.
  - `create_request()` performs the balance check and registers a hold (guarded against duplicates).
  - `decline_request()` and `cancel_request()` release holds.
  - `mark_complete()` records each participant's completion mark and, once both confirm, **delegates settlement to `CreditRepository.settle_exchange_credits()`** â€” no inline raw SQL or business logic in the controller, per `guide.md` Â§5/Â§10.
- `app/routes/frontend_routes.py` maps `/credits/ledger` and the exchange completion POST to the controller.
- `app/templates/credits/ledger.html` displays available balance, reserved credits, the full transaction history (date, type, linked exchange, signed credit change), and a "how credits move" panel. It contains **no** top-up, credit-card, PayPal, or payment UI.

## Database and Security Notes

- Atomic double-entry settlement: `settle_exchange_credits()` runs debit, credit, ledger inserts, hold clearing, and count increments inside one `db.transaction()`, so they succeed or fail together as required by `guide.md` Â§10 (credit transfers must be atomic).
- Idempotency: `exchange_already_settled()` plus an in-transaction re-check means refreshing or resubmitting the completion form never moves credits twice.
- Duplicate-hold prevention: `has_active_hold_for_request()` stops a second active hold being created for the same request from a double-submitted form.
- Hold release guarantee: declining or cancelling a request sets the hold to `released`, restoring the learner's available balance.
- No real money: the ledger UI exposes only internal credits; there is no top-up, payment, deposit, or withdrawal path anywhere in the codebase.
