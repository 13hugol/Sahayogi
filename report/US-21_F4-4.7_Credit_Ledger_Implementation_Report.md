As a user, I want a credit ledger system so that I can earn credits by teaching and spend them to learn.

## Completion Status

Completed for both backend and frontend wallet/ledger interfaces on the `main` branch. Users start with 100 default credits. Credit holds (reservations) are automatically managed during skill request lifecycles. Once an exchange is marked completed, the credit hold is settled: learner balance is debited and teacher balance is rewarded, with corresponding ledger transaction entries recorded in MySQL.

## Implementation Summary

- `app/database.py` defines schema structures and Alter table migrations for `users.credit_balance`, `credit_transactions`, and `credit_holds`.
- `app/models/credit_transaction.py` defines `CreditTransaction` and `CreditHold` models.
- `app/models/user.py` loads `credit_balance` and calculates `available_credit_balance` by subtracting active holds.
- `app/repositories/credit_repository.py` handles database queries for holds, releases, and transaction records.
- `app/controllers/frontend_controller.py`:
  - `wallet()` retrieves history and active holds to render `ledger.html`.
  - `create_request()` performs credit checks and registers holds.
  - `decline_request()` and `cancel_request()` release holds.
  - `mark_complete()` completes exchanges, releases holds, transfers credits, and inserts double-entry ledger records.
- `app/routes/frontend_routes.py` routes exchange completion POST requests to `FrontendController.mark_complete`.
- `app/templates/credits/ledger.html` displays dynamic balance, reserved holds, and recent transactions.

## Acceptance Criteria Evidence

| Acceptance Criterion | Implementation Evidence |
| --- | --- |
| Users earn a configurable number of credits, default 10, upon successful completion of a teaching session. | When an exchange is marked complete, `mark_complete` transfers the credit cost defined on the skill listing (which defaults to 10) to the teacher's balance and records an `earning` transaction. |
| Requesting a skill from a one-sided listing deducts the required credits from the learner's balance. | A request triggers `CreditRepository().create_hold()` which reserves the listing's cost. This reduces the learner's available balance immediately. On exchange completion, this hold is settled and debited as a `deduction` transaction. |
| Ledger page shows a full transaction history with date, action, skill, and credit change for each entry. | `/credits/ledger` fetches and renders all `credit_transactions` and active `credit_holds` for the logged-in user in chronological order. |
| System prevents a user from initiating a credit-based request if their balance is insufficient. | `FrontendController.create_request` checks `current_user.available_credit_balance < listing.credit_cost`. If true, the request is rejected and redirected with flash feedback. |

## Database and Security Notes

- Double-Entry Ledger Protection: Credit transfers and status updates are wrapped in a database-level transaction (`with db.transaction()`) to guarantee that learner debit and teacher credit succeed or fail as a single unit.
- Hold Release Guarantee: If a request is declined by the provider or cancelled by the learner, the hold is safely set to `released`, restoring the learner's available credit balance.
