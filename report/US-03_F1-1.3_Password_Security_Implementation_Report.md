As a user, I want my password to be securely hashed so that my credentials are protected even if the database is compromised.

## Completion Status

Implemented on the `Chayanga` branch for the current Flask/Jinja backend scope. The story now includes secure password hashing, one-time password reset tokens with 30-minute expiry, reset email delivery through the existing email utility, and a profile-settings password change form that requires the current password.

## Implementation Summary

- Added `app/utils/passwords.py` to centralize secure password hashing, password verification, and reset-token hashing.
- Added `PASSWORD_HASH_METHOD` configuration using Werkzeug `scrypt` hashing, an accepted bcrypt-equivalent adaptive password hash.
- Added the `password_reset_tokens` table with hashed token storage, expiry, used-state tracking, and user cascade cleanup.
- Extended `AuthService` and `UserRepository` with password reset token creation, validity checks, one-time consumption, and password replacement.
- Replaced the placeholder forgot-password handler with a real `/auth/forgot-password` flow.
- Added `/auth/reset-password/<token>` for one-time password resets.
- Added `/auth/change-password` for authenticated profile-settings password changes after current-password confirmation.
- Updated `auth/forgot_password.html`, `auth/reset_password.html`, and `profile/edit.html` for the user-facing password security flow.
- Added focused US-03 automated tests in `tests/test_auth.py`.

## Acceptance Criteria Evidence

| Acceptance Criterion | Implementation Evidence |
| --- | --- |
| Passwords are hashed using bcrypt or equivalent with a minimum cost factor of 10 before storage. | `hash_password()` uses configured Werkzeug `scrypt` hashing before user creation, reset, or password change. Stored hashes are verified through `verify_password()`, never by plain-text comparison. |
| Plain-text passwords are never stored or logged anywhere in the system. | Only password hashes are persisted in `users.password_hash`; reset emails contain only a one-time reset URL and tests assert the submitted password is not written to the mail log. |
| Password reset generates a one-time secure token sent via email, expiring within 30 minutes. | `AuthService.request_password_reset()` creates a high-entropy token, stores only its SHA-256 hash in `password_reset_tokens`, sets expiry from `PASSWORD_RESET_EXPIRY_SECONDS` (`1800` seconds), and sends the URL via `send_email()`. `consume_password_reset_token()` marks it used during password replacement. |
| Users can change their password from profile settings by confirming their current password first. | `profile/edit.html` now includes a Password and security form. `/auth/change-password` validates the current password before replacing the hash and flashes an error if confirmation fails. |

## Database and Security Notes

- Reset tokens are not stored in plain text; only SHA-256 token hashes are persisted.
- Creating a new reset token invalidates any previous unused token for the same user.
- Password replacement clears failed-login counters and lockout state after a successful reset or change.
- Reset responses do not reveal whether an email address belongs to an account.
- Existing CSRF protection applies to forgot-password, reset-password, and change-password POST requests.
