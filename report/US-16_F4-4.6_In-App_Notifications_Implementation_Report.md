## Completion Status

Implemented on the `Chayanga` branch for the current Flask/Jinja backend scope. The story now has a persisted notification model, event-specific notification generation service, notification panel, unread notification count endpoint, individual read handling, and a `Mark all as read` action.

## Implementation Summary

- Added `Notification` as the model for bell notifications with event type, title, body, target URL, read state, and timestamp fields.
- Added the `notifications` database table with user ownership, read-state indexes, event-type indexing, and cascade cleanup when a user is deleted.
- Added `NotificationRepository` with parameterized SQL for create, list, unread count, scoped lookup, individual mark-read, and mark-all-read operations.
- Added `NotificationService` helpers for the required event types: new match, exchange request, accepted request, declined request, new message, and new review.
- Updated the notification page so it loads the signed-in user's notifications from persistence and shows timestamped notification links.
- Updated notification routes so opening a notification marks only that user's notification as read and redirects to a safe internal target URL.
- Updated the `Mark all as read` route to validate CSRF and mark only the signed-in user's unread notifications.
- Updated the navigation count endpoint so the existing JavaScript badge polling returns the persisted unread notification count while leaving unread message badge behavior untouched.
- Updated the dashboard recent-notification area and unread notices metric to use persisted notifications.

## Database and Security Notes

- All notification SQL uses `%s` placeholders and repository methods keep user scoping in every read/update query.
- Notification target redirects are constrained to internal relative URLs to avoid open redirects.
- Notification POST actions remain protected by the existing CSRF flow.
- This implementation intentionally does not implement or alter the separate US-15 unread message badge story; message counts remain unchanged.
- Adjacent exchange, messaging, review, and match workflows are still frontend-shell routes in the current codebase, so the notification service is the scoped integration point for those event producers when those stories become active.
