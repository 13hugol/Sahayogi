# US-26 / F5-5.4 - Role-Based Access Control Implementation Report

## User Story

As an admin, I want role-based access control so that sensitive actions are protected based on user role.

## Completion Status

Completed on the `main` branch. Admin routes are protected server-side, new users receive the default `user` role, existing admins can change another user's role, and admin-only links are hidden from non-admin users.

## Implementation Summary

- `app/auth.py` defines `admin_required`.
- `app/__init__.py` configures Flask-Login and loads users from MySQL.
- `app/models/user.py` defines `Role`, `User.is_admin`, default user creation, and role update behavior.
- `app/controllers/admin_controller.py` uses `admin_required` on every admin controller method.
- `app/routes/admin_routes.py` maps admin-only endpoints.
- `app/templates/base.html` only shows the Admin navigation link when `current_user.is_admin` is true.
- `app/templates/admin/users.html` lets admins promote or demote other users between `user` and `admin`.

## Acceptance Criteria Evidence

| Acceptance Criterion | Implementation Evidence |
| --- | --- |
| All admin routes return 403 for non-admin users. | `admin_required` aborts with 403 when the current user is missing the admin role. |
| Role defaults to `user` at registration. | `AuthController.register()` calls `Role.ensure("user", ...)` before creating a registered user. |
| Role can only be elevated by an admin. | `AdminController.update_user_role()` is protected by `admin_required` and rejects self-role changes. |
| Sensitive endpoints validate role on every request. | Admin controller methods are decorated individually, so each request checks the current user's role. |
| Frontend hides admin links for regular users. | `base.html` renders the Admin link only when `current_user.is_admin` is true. |
| Backend enforces the same restriction independently. | Browser and automated tests confirm direct `/admin/` access by a regular user returns 403. |

## Database and Audit Notes

- Roles are stored in the `roles` table.
- Users reference roles through `users.role_id`.
- Admin role changes create an `admin_audit_logs` record with the acting admin, target user, and detail text.
