## Implementation Summary

- `app/database.py` includes `suspended_until` and `suspension_reason` fields and handles dynamic database migration.
- `app/enums.py` registers the new `AccountStatus` values.
- `app/models/user.py` maps the new fields in constructor and `from_row()`.
- `app/repositories/user_repository.py` handles database-level status updates using `update_status`.
- `app/repositories/skill_repository.py` handles deactivating a banned user's listings using `deactivate_all_for_user`.
- `app/services/auth_service.py` blocks login attempts of restricted users with clear descriptive messages.
- `app/__init__.py` implements a global `before_request` hook (`check_user_status`) that catches active sessions, logs out suspended/banned users immediately, pops CSRF tokens, and displays details.
- `app/services/admin_service.py` coordinates suspension, ban, and lift actions and logs them to `admin_audit_logs`.
- `app/controllers/admin_controller.py` maps the POST requests and handles feedback messages.
- `app/routes/admin_routes.py` exposes the suspend/ban endpoints.
- `app/templates/admin/users.html` adds a custom Account Enforcement pane with validation inputs.

## Acceptance Criteria Evidence

| Acceptance Criterion | Implementation Evidence |
| --- | --- |
| Admin can apply a temporary suspension (1-30 days) or a permanent ban from the user management panel. | The `users.html` template displays an Account Enforcement pane with forms to suspend (1-30 days duration input and reason) or ban (reason) any selected user. |
| Banned/suspended users are immediately logged out and see a clear message explaining their account status. | A global `before_request` hook catches any authenticated restricted user, terminates the session via `logout_user()`, and redirects with a custom explanation. Attempting to log in raises an exception with the exact suspension end time and reason. |
| All active listings belonging to a banned user are automatically deactivated. | Banning a user calls `SkillRepository().deactivate_all_for_user()`, changing all their active listings status to `deactivated` in MySQL. |
| Admin must enter a reason for the ban, which is stored in the audit log and optionally sent to the user. | Ban and suspend forms require a reason. The reason is written to `admin_audit_logs` target details and displayed on the login page as flash feedback. |

## Database and Security Notes

- Temporary suspension automatic lifting: If a user's suspension period has expired, the first authentication or request check automatically lifts the restriction and restores active status in the database.
- Self-enforcement protection: UI and backend prevent admins from suspending or banning themselves.
