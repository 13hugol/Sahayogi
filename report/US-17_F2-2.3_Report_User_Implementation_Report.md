As a user, I want to report another user so that the platform remains safe and respectful.

## Completion Status

Completed on the `Prashna` branch. The system now supports a user reporting mechanism allowing users to report others from their public profiles, preventing self-reporting, limiting submissions against the same user to once every 7 days, notifying the reporter upon receipt, and allowing administrators to review and resolve/dismiss reports with proper audit logging.

## Implementation Summary

- **Database Updates**: Added the `reports` table in `app/database.py` with foreign keys referencing the users table, and updated the table clearing sequence in `tests/conftest.py`.
- **Model Classes**:
  - `Report` model dataclass in `app/models/report.py` with helper properties to dynamically look up the reporter and reported user objects.
  - Exposed in `app/models/__init__.py`.
- **Repositories**:
  - `ReportRepository` in `app/repositories/report_repository.py` to handle CRUD operations on reports, including a duplicate check (`has_recent_report`) inside a 7-day window.
  - Exposed in `app/repositories/__init__.py`.
- **Service Layer**:
  - Integrated `list_reports` and `resolve_report` in `AdminService` (`app/services/admin_service.py`) to manage and persist administrator decisions, logging audit entries.
- **Controllers & Routing**:
  - Handled the submit action in `FrontendController.report_user` with validation (duplicate check, reason choices, and self-report check).
  - Implemented the dashboard viewing and resolution actions in `AdminController` (`app/controllers/admin_controller.py`).
  - Registered resolve route in `app/routes/admin_routes.py`.

## Acceptance Criteria Evidence

| Acceptance Criterion | Implementation Evidence |
| --- | --- |
| A `Report User` button is accessible from any user's public profile page. | The profile layout in `app/templates/profile/view.html` displays the report form with reason and description fields when viewing another user's profile while logged in. |
| Report form requires selecting a reason: spam, harassment, fake profile, fraud, or other; description is optional. | Extends `ReportShellForm` with selectable reasons (`spam`, `harassment`, `fake_profile`, `fraud`, `other`) and a textarea description. Validated backend-side in `FrontendController.report_user`. |
| Submitted reports are sent to admin review; reporter receives confirmation that the report was received. | Created reports are persisted with status `'open'`. The reporter receives an in-app notification: *"Your report against <reported user> has been received and is under review."* |
| A user cannot submit duplicate reports against the same user within a 7-day window. | Verified in `ReportRepository.has_recent_report` which checks if a report was submitted in the last 7 days. Enforced in `FrontendController.report_user`. |

## Database and Security Notes

- **Access Controls**: The report submission endpoint uses `@login_required`. The administrative dashboard and resolution endpoints use `@admin_required`.
- **Audit Trails**: Resolving or dismissing a report generates a detailed log in `admin_audit_logs`.
- **CSRF Protection**: All POST request forms validate CSRF tokens via Jinja.
