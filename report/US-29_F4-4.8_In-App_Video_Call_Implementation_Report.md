## Completion Status

Completed for both backend controllers and the interactive frontend. The video call is a **real in-app WebRTC peer call**: camera and microphone media flow peer-to-peer through `RTCPeerConnection`, while signaling (offer, answer, ICE candidates, and a `leave` notice) is exchanged through lightweight MySQL-backed JSON endpoints. No external paid calling provider is used; the implementation stays within Flask, MySQL, Jinja, Bootstrap, and vanilla JavaScript as required by `guide.md` §3. Call state in the database is authoritative: a call can only start on an active exchange between two participants who are both online, and ending a call clears state, logs a session summary, and notifies the other peer to hang up. Presence heartbeats drive the 2-minute reconnect countdown.

## Implementation Summary

- `app/database.py` defines the `video_call_signals` table (`exchange_id`, `sender_id`, `recipient_id`, `signal_type`, `payload`, `consumed_at`, `created_at`) plus the existing user-activity and active-call columns.
- `app/models/exchange.py` carries call state: `video_call_active`, `video_call_started_at`, `video_call_ended_at`, and `video_session_summary`.
- `app/models/video_call_signal.py` defines the `VideoCallSignal` model.
- `app/models/user.py` exposes `is_online` (active within 2 minutes).
- `app/repositories/exchange_repository.py` handles starting/ending a call and finding the active exchange between two users.
- `app/repositories/video_call_signal_repository.py` stores signaling messages, returns and consumes pending signals for a participant, and clears signals when a call ends.
- `app/controllers/frontend_controller.py`:
  - `_resolve_video_call_exchange()` centralizes participant access control for every video endpoint (404/403).
  - `video_call_room()` renders the room and exposes the deterministic offerer role and ICE server config to the template.
  - `start_video_call()` is authoritative: only on an active exchange, only for participants, and only when both are online.
  - `end_video_call()` clears call state, drops stale signals, posts a `leave` signal to the peer, computes duration, logs the summary, and posts a chat message.
  - `video_call_heartbeat()` updates presence and returns peer-online/call-active/summary state.
  - `video_call_signals()` is a participant-only `GET`/`POST` JSON endpoint: `GET` returns and consumes unconsumed signals for the caller; `POST` stores an `offer`/`answer`/`ice`/`leave` signal for the other participant.
- `app/__init__.py` extends CSRF protection to accept the `X-CSRFToken` header so JSON signaling requests are protected.
- `config.py` adds `WEBRTC_ICE_SERVERS`, defaulting to an empty list (no external dependency) with optional `WEBRTC_ICE_SERVERS` env override for deployment behind NAT.
- `app/routes/frontend_routes.py` maps `/exchanges/<id>/video/signals` (GET/POST) alongside the existing start/end/heartbeat endpoints.
- `app/templates/exchanges/video.html` implements a genuine `RTCPeerConnection`: local camera/mic capture, remote media via `ontrack`, ICE exchange through polling, mic/camera toggles, screen-share via `replaceTrack`, remote-ended handling through the `leave` signal, and the existing 2-minute reconnect countdown.
- `app/templates/messages/detail.html` and `app/templates/exchanges/detail.html` both surface a Start/Join Video Call control whenever an active exchange exists, so the option is always discoverable.

## Database and Security Notes

- Participant-only access: `_resolve_video_call_exchange()` enforces that the caller is the learner or teacher of the exchange on every video endpoint; non-participants get `403`.
- Authoritative call state: calls require an `active` exchange and both participants online to start; ending clears `video_call_active` and stale signals.
- CSRF: JSON signaling `POST` requests carry the `X-CSRFToken` header, validated by the application's CSRF protection.
- Signaling hygiene: consumed signals are marked so they are not replayed; ending a call deletes pending signals to keep the next call clean.
- No external dependency by default: `WEBRTC_ICE_SERVERS` defaults to empty, so local two-party calls work without a STUN/TURN provider; deployments can configure one via the environment.
